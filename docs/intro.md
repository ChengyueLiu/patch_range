# Introduction

第三方库和上游项目代码在 C/C++ 生态中被广泛复用——通过 fork、vendor、static linking 等方式，一份上游代码可以被分发到成百上千个下游项目和版本中。当上游披露一个 CVE 时，安全响应方需要回答的第一个问题是：这个漏洞影响了哪些版本？这个回答决定了补丁分发的范围、合规审计的判定、以及下游用户的应急响应优先级。然而，NVD 等公开漏洞数据库提供的 affected version 信息往往是粗粒度的版本区间，且大量条目存在标注错误或缺失，无法直接作为决策依据。准确、自动化地识别一个 CVE 影响了哪些版本（vulnerability-affected version identification），因此成为软件供应链安全中的一项基础任务。

围绕这个任务，学术界发展出两类方法。Matching 类方法（如 ReDeBug、VUDDY、MOVERY、V1SCAN、FIRE）从 fixing patch 提取漏洞特征（代码片段、函数哈希、token n-gram），然后在每个目标版本中搜索是否存在该特征；存在即判该版本受影响。Tracing 类方法（如 SZZ、V-SZZ、Lifetime、SEM-SZZ、LLM4SZZ）则从 fixing commit 出发，沿 git 历史回溯找到引入漏洞的 commit，再根据 tag 拓扑推断哪些版本含该 commit 但未应用 fix。两类方法各有侧重：matching 直接、对版本间代码相似性敏感；tracing 不依赖代码字面匹配、对代码演化更鲁棒。然而，ASE 2025 上一项系统性 study 在 1128 个 CVE / 9 个 C/C++ 项目上评估 12 个代表性工具，得出一个令人意外的结果：没有任何单一工具的 accuracy 超过 45%；最佳 F1 不超过 78%；即使将多个工具 ensemble，accuracy 仍低于 60%。这个数字表明：单纯叠加更多 syntactic 启发式难以突破现状——问题需要从更底层重新审视。

我们对这 12 个工具的失败案例做系统性根因分析后发现，问题不在于哪个工具做得不够细，而在于它们共享了一个根本设计选择。考虑这样一个例子：CVE-2022-4203 的 fix 改动了 OpenSSL 的 X.509 name constraint 校验逻辑，针对的是 RFC 8398 引入的 SmtpUTF8Mailbox 类型——这是 OpenSSL 3.0 才有的 feature。然而 OpenSSL 1.0.0 已经存在这个文件（虽然目录路径不同），只是文件里完全没有 SmtpUTF8 相关代码。现有工具看到"1.0.0 含该文件、且未应用 fix commit"就把 1.0.0 标为受影响——但 1.0.0 上根本不存在漏洞。这类错误不是工具的实现缺陷，是因为现有工具用 syntactic 信号（文本匹配、commit hash、git 拓扑）同时完成两个本质不同的子任务：在历史版本里**定位**与 fix 相关的代码段（locate），和**判断**这段代码是否真的处于 vulnerable 状态（judge）。把两件事压到一个 syntactic paradigm 上做，必然在三类场景下失败。第一，对于本可由更合适的 syntactic 策略解决的 case，现有工具普遍策略选错：22.6% 的 CVE 是 ADD-only fix（fix 只加防御代码、不删任何代码），但现有工具几乎全部以 deleted lines 作为追溯起点，对 ADD-only 直接失效。第二，当 locate 需要在目标版本里找到经过重构或跨文件迁移的代码（如 FFmpeg 中 applehttp.c 改名为 hls.c 并伴随代码重写），文本相似度无法捕捉语义同源，trace 路径中断。第三，如开头 OpenSSL 例子所示，即使 locate 成功，syntactic 信号也无法判断"目标版本是否真的 vulnerable"——这要求理解漏洞机理。三类失败彼此独立、累积叠加，构成了 study 观测到的 "<45% accuracy" 的统一根因。这也意味着改进方向不在于设计更巧的 syntactic 启发式，而在于把 syntactic 限于它真正擅长的边界（处理拓扑、做粗粒度匹配），把 locate 和 judge 中真正需要语义理解的部分让位给语义方法。
