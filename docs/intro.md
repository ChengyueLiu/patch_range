# Introduction

软件供应链安全中，准确识别 CVE 披露的漏洞影响范围至关重要：它直接决定了补丁分发的范围、合规审计的判定、以及下游应急响应的优先级。然而要准确做到这一点并不容易：需要追踪代码在不同版本间的演化（包括重构、迁移、分支差异等），并把这些演化与漏洞特征对应起来，单靠规则匹配难以胜任。这种难度的代价反映在数据上：He et al. [TDSC'24] 在 Linux Kernel 上的系统测量发现，即便是 NVD 这样权威的公开漏洞数据库，其 affected version 标注也存在 5.7% 假阳性和 21.3% 假阴性。这种不准确性的影响在 C/C++ 生态中尤其严重：大量底层基础设施（如操作系统内核、加密库、媒体处理库）都用 C/C++ 编写，单个上游 CVE 可能波及成百上千个下游项目和版本，标注一旦错误，影响面被显著放大。因此，准确、自动化地识别一个 CVE 影响了哪些版本（vulnerability-affected version identification），成为软件供应链安全研究的一项核心任务。

现有的方法主要分为两类：**Matching 类方法**（如 ReDeBug、VUDDY、MOVERY 等）从 fixing patch 出发，主要靠**代码内容的相似度匹配**：把"漏洞代码长什么样"编码成一组可搜索的特征（如函数级哈希、token n-gram、或抽象后的代码片段），然后在每个目标版本里搜索这些特征是否存在，存在即判该版本受影响。**Tracing 类方法**（如 V-SZZ、SEM-SZZ、LLM4SZZ 等）则换一个角度，主要靠**git 历史的反向追溯**：先在 fix patch 里挑出能代表漏洞的"目标行"（现有工作普遍用 fix 删除的代码行作为锚点），再用 git blame 等工具沿历史反向走，找到最早把这些行写进 repo 的 commit（即 intro commit），最后通过 git tag 拓扑找出含 intro commit 但未含 fix commit 的版本，把这些版本判为受影响。两类方法的本质差异在于：matching 在代码层做相似度匹配，tracing 在 git 历史层做 commit 追溯。尽管机理不同，两类方法在最终判定阶段都依赖 syntactic 信号（文本匹配、commit hash、git 拓扑），不涉及对漏洞或代码语义本身的理解。

尽管现有的 matching 与 tracing 方法已能识别相当一部分漏洞影响范围，整体准确性仍然有限。最新一项大规模 empirical study 在 1128 个 CVE 上评估了 12 个代表性工具，结果是没有任何一个工具能可靠地识别受影响版本：即便表现最好的工具 accuracy 也不到 45%，将多个工具组合起来也仅略有改善。我们对这些工具的方法、源代码及失败案例进行了综合分析，发现 syntactic 信号作为代理在三种场景下系统性失败。**(1) 没有可追溯的锚点**：占 22.6% 的 CVE 是 ADD-only fix（fix 仅增加防御代码、不删任何代码），syntactic 缺少 deleted lines 作为追溯起点。**(2) 追溯过程断裂**：代码在 V 上被重构或跨文件迁移（如 FFmpeg 的 applehttp.c 改名为 hls.c，并伴随代码重写），文本相似度无法捕捉语义同源。**(3) 追溯到了但无法判断**：例如 CVE-2022-4203 的 fix 改动的代码涉及 OpenSSL 3.0 才引入的一个新功能，但 OpenSSL 1.0.0 上已经存在 fix 改动的同名文件（只是文件里没有那个功能的代码），现有工具看到"1.0.0 含该文件、不含 fix"就把它误判为 affected。事实上，这三类失败对应了 trace 流程中三个关键决策点：*anchor*（在 fix patch 里挑选哪段代码作为追溯起点）、*propagate*（追溯过程中代码语义发生变化时如何识别同源），以及 *judge*（追溯到目标版本后判断该版本是否真的存在漏洞）。现有方法在每个决策点都用 syntactic 信号代理决策，但 *propagate* 与 *judge* 本质上是关于代码语义与漏洞机理的判断：前者要求识别"语义同源但文本已经发生重构、迁移、改名的代码"，后者要求理解"漏洞特征是否在该版本上真实成立"。文本与拓扑本身不携带这两类信息，syntactic 代理因此天然存在能力上限。

要让 *propagate* 与 *judge* 真正得到处理，必须引入能在代码层面理解漏洞机理与语义同源关系的能力。LLM 与 AI Agent 的成熟使这一能力首次成为可能：已有工作（如 AgentSZZ、SZZ-Agent）让 agent 自主调用 git 工具完成 trace 类任务的语义推理，验证了这一可行性。然而，将整个 affected version identification 流程交由 agent 端到端主导并不现实。其一，agent 的决策过程是黑盒，长链工具调用中可能产生幻觉（如"声称在某版本找到了某段代码"而该代码实际不存在），且错误难以追溯与复现，使方法在实际安全场景中难以审计；其二，每个 case 数十次 LLM 调用的开销也限制了它在 1000+ CVE × 数百候选版本规模下的可用性。我们因此需要一种方法：既能让 agent 在 *propagate* 与 *judge* 这两个真正需要语义判断的决策点上发挥作用，又能把它的不确定性与开销约束在最小范围，其余决策与流程协调由确定性算法承担——这样既保留 agent 的语义能力，又让方法整体可审计、可复现、可在真实安全响应场景中规模化部署。
